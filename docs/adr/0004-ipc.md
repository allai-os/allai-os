# ADR-004: IPC entre componentes

- **Estado**: Aceptado
- **Fecha**: 2026-04-28
- **Decididores**: Juan Manuel Castellanos (BDFL)

## Contexto

allAI OS tiene múltiples procesos que cooperan:

- `allaid` (Rust, daemon systemd-user)
- Agent core (Python, lanzado por `allaid` en sandbox)
- Overlay UI (Python/GTK4)
- GNOME Shell extension (GJS)
- CLI `allai-ctl` (Rust)
- Posibles clientes externos (apps de terceros que quieran integrar allAI)

Necesitamos un mecanismo de IPC que sea:

- Estándar en Linux desktop.
- Soportado en todos los lenguajes de la pila.
- Capaz de exponer una API tipada (introspección).
- Compatible con polkit para autorización.
- Capaz de propagar eventos (señales pub/sub).
- Auditable.

## Decisión

Usaremos **D-Bus** como el bus de IPC primario. `allaid` expondrá la interfaz `org.allai.Agent1` en el bus de sesión del usuario.

Detalles:

- **Bus**: session bus (`$DBUS_SESSION_BUS_ADDRESS`). El agente vive en el espacio del usuario; nada corre como root.
- **Servicio bien-conocido**: `org.allai.Agent1`.
- **Object paths**: `/org/allai/Agent1`, `/org/allai/Agent1/Session/N`, `/org/allai/Agent1/AuditLog`.
- **Introspección**: XML estándar D-Bus + tipos documentados.
- **Eventos**: señales para `ActionStarted`, `ActionCompleted`, `PermissionRequested`, `KillSwitchTriggered`, etc.
- **Polkit**: acciones que requieran elevación van por polkit, no por sudo.
- **Política**: archivo `org.allai.Agent1.conf` en `/usr/share/dbus-1/session.d/` define quién puede llamar qué.

Para datos voluminosos (screenshots, embeddings, audio):

- **Memoria compartida**: `memfd_create` + paso de fd por D-Bus para buffers grandes (screenshots).
- **Sockets Unix**: para streams de audio/video largos (PipeWire ya lo hace; reutilizamos su patrón).

## Alternativas consideradas

- **gRPC sobre socket Unix**: protocolo moderno, tipado fuerte, multi-lenguaje. Pero no integra con polkit ni con el ecosistema de portales/systemd. Reinventar lo que D-Bus ya da.
- **HTTP/JSON-RPC en localhost**: simple, pero pierde tipos, requiere autenticación propia, no integra con polkit.
- **Sockets brutos con protocolo propio**: máxima eficiencia, máximo trabajo, mínima interoperabilidad. Descartado.
- **Sólo IPC entre procesos hijos (pipe + JSON)**: insuficiente, no permite que clientes externos (CLI, extensiones, scripts del usuario) hablen con el agente.
- **Varlink**: alternativa moderna a D-Bus (systemd la usa). Más simple pero menos extendida; descartada por madurez del ecosistema.

## Consecuencias

### Positivas

- Estándar de facto en Linux desktop. Todo el mundo lo entiende.
- Bindings de calidad en Python (`dasbus`, `pydbus`), Rust (`zbus`), JS (Gio.DBus).
- Polkit integra de forma natural con D-Bus.
- Introspección facilita herramientas de debug (`busctl`, `d-feet`, `d-spy`).
- Permite que apps externas integren allAI sin que tengamos que publicar SDKs propietarios.

### Negativas

- D-Bus tiene reputación de complejidad. Documentar bien las interfaces ayuda.
- Datos grandes (>1MB) se vuelven costosos por D-Bus → la regla "memfd o socket aparte" es importante.
- Versionar interfaces requiere disciplina (numerar interfaces: `Agent1`, `Agent2`).

### Neutras

- Necesitaremos un esquema de versionado claro de la API D-Bus (semver-like en interfaces).

## Plan de implementación

1. Definir XML de la interfaz `org.allai.Agent1` en `system/allaid/dbus/org.allai.Agent1.xml`.
2. Implementar servidor en `allaid` con `zbus` (Rust).
3. Implementar cliente Python con `dasbus` para el agent core hablando con `allaid`.
4. Implementar cliente Rust en `allai-ctl`.
5. Implementar cliente GJS en la extensión GNOME.
6. Tests con `dbus-launch` en CI.
7. Documentar API en `docs/api/dbus.md`.

## Revisión

Reevaluar si:

- Aparece carga de IPC que D-Bus no maneje bien (improbable con la arquitectura actual).
- systemd/Varlink desplaza a D-Bus en upstream Fedora (poco probable a corto plazo).

Plazo de revisión: anual.
